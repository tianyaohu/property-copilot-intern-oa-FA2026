## Design decisions

1. **Map provider** — which provider you chose and the trade-off versus the alternatives.
   - MapLibre GL via react-map-gl, styled with OpenFreeMap's free `liberty` vector style: no API key or billing account and all are open for commercial use. MapLibre offers smoother pan/zoom at scale with WebGL. OpenFreeMap also offers rich 3D geographical data when zoomed in, allowing users to checkout nearby amendities, like parks, stores, and transit stops. 
   
2. **Performance at density** — how you keep the map smooth with all markers visible (clustering, viewport rendering, etc.) and what you observed.
   - Considered clustering and declined it at n=50: fifty declaratively-rendered marker pills are trivial for the DOM (pan/zoom stays smooth with all markers visible), and cluster bubbles would hide the "every marker visible" behaviour the product wants. Threshold: at ~300+ overlapping markers I'd add clustering or canvas/GeoJSON-layer rendering; viewport-driven fetching already bounds what renders.
   - The measured bottleneck was the server, not the map: the initial metro view fans out to 104 geo-index partitions, and at 256 MB the Lambda spent 0.9–1.5 s in-function dispatching those Queries (CPU-bound TLS + signing — the same query is 7 ms for a 4-partition downtown view). Raising the Lambda to 1024 MB (CPU scales with memory) cut it to 0.2–0.33 s, ~0.35–0.5 s end-to-end warm.
   - Guards sized from measurement: a realistic full-Metro-Vancouver viewport estimates to ~726 partitions, comfortably inside the 800 cap. The map itself no longer caps how far a user can zoom out, so a genuinely world-scale request is expected and intended to hit this guard — surfacing as "Viewport too large; zoom in" rather than being pre-emptively blocked by map bounds. The estimate runs *before* cell enumeration, since a world-scale box would otherwise allocate ~33M strings; per-partition pagination and a 200-item cap protect correctness and payload size at real data volumes. The cap applies *after* attribute filters (so a narrow filter never loses matches to it), and the response's `truncated` flag surfaces in the UI as a "zoom in to see all" hint.
3. **Geospatial querying** — how the server answers a viewport query without scanning the table, and how you use the geohash index.
   - At write time, each listing derives both a full-precision geohash and a 5-char prefix (~5km cell); the prefix is the GSI partition key, so each Query is naturally scoped to one cell. On a viewport query, ngeohash.bboxes enumerates every overlapping prefix cell, each fired as a parallel QueryCommand against geo-index. Results are then refined to the exact bounding box and deduplicated — keeping read cost proportional to viewport size, never table size.


4. **Filtering model** — the dimensions you support, how filters compose, and the empty/reset behaviour.
   - Four dimensions — rent range (inclusive), bedrooms, bathrooms (the extra dimension), property type — all parsed and applied server-side by `filter.ts`; they AND together after the geospatial query, so one request answers viewport + filters and the map and list can never disagree. Query parsing rejects blank values explicitly (`?maxRent=` is ignored rather than coerced to `0` by `Number("")`).
   - Filtering happens in app code post-refine rather than a DynamoDB `FilterExpression`: Query pricing is per item read, not returned, so pushing predicates down saves only network transfer on a candidate set that is already viewport-sized — while app-side keeps the logic pure, shared, and unit-testable.
   - The filter UI is an Airbnb-style pill bar: quick-toggle pills (property type, "1+ bathrooms") apply instantly, and "Price"/"Filters" pills (the latter with an active-count badge) open a modal — rent range plus bedroom/bathroom steppers — that stages edits until "Apply", with an instant "Clear all" as the reset. Empty results keep the map interactive and show an empty state in the list; filter edits share the viewport debounce, so rapid changes coalesce into one request.

## What I'd add with more time

- **A DOM test environment for the frontend** (jsdom + Testing Library): the pill bar, modal draft/cancel semantics, and the browse page's debounce/abort flow are only covered manually today — the repo deliberately carries no DOM test dependency, so component behaviour is untested in CI.
- **A pagination assertion for the per-prefix `LastEvaluatedKey` loop**: exercising it honestly needs >1 MB of items in a single ~4.9 km geohash cell, disproportionate to seed for an OA; I'd assert it with a mocked DynamoDB client instead.
- **Cursor-based "load more" past the 200-item cap** (the API already reports `truncated`; the UI currently says "zoom in to see all"), and marker clustering once listing density crosses the ~300-marker threshold above.
- **Antimeridian-aware viewport splitting**: out-of-range longitudes from extreme zoom-outs are clamped client-side, which collapses an antimeridian-crossing view into one wide box — fine for a Metro Vancouver product, but a global one should split the box in two.
